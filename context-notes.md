# 개발 컨텍스트 및 결정 의사록

개발 중에 내린 결정과 그 배경 논리를 기록하는 문서입니다.

## 결정 사항

### React Native Expo 프레임워크 선정
- 네이티브 모듈 설정이 간단하고 로컬에서 즉시 시뮬레이션 및 웹 테스트가 가능한 Expo 프레임워크를 선택하였습니다.
- TypeScript를 도입하여 API 데이터 모델링의 안전성을 극대화합니다.

### 스타일링 방식에 대한 논의
- 복잡한 NativeWind 빌드 설정 대신 안정성과 커스텀 편의성을 우선하여 표준 React Native StyleSheet를 사용해 정교한 Pearl White 테마의 디자인을 구축합니다.

### 로컬 Uvicorn 통신 IP 설정 방식
- 시뮬레이터와 물리 기기에서의 연결성을 위해 앱 실행 시 기본 API 주소를 유연하게 전환할 수 있도록 처리합니다.

### Firebase 연동 프로젝트 정보 설정
- 사용자가 제공한 실제 프로젝트 ID인 action-b8c75를 설정 파일에 반영하여 실질적인 데이터 동기화 기반을 마련하였습니다.

### UI 여백 축소 및 모바일 시군구 필터 전국 확장
- PC 대시보드의 글꼴 배율 축소(120% -> 100%) 및 주요 패널(서랍, 사이드바 등)의 padding/margin 축소를 통해 빈 공간이 보이지 않도록 UI 밀도를 조절하였습니다.
- 모바일 앱의 시도 필터 대상을 5개에서 17개 전국 단위로 확장하고, 칩 컴포넌트들이 영역을 과도하게 차지하지 않도록 ScrollView horizontal 기반 가로 스크롤 레이아웃을 도입하였습니다.

### 온라인 연동 검증 및 아키텍처 문서화 방향 결정 (2026-06-06)
- PC 대시보드(index.html)와 모바일 앱은 Firestore 연동을 통해 100% 온라인 직접 조회가 정상적으로 수행됨을 확인하였습니다.
- 그러나 신규 데이터를 주기적으로 수집하여 Firestore로 마이그레이션하는 크롤러 및 DB 엔진(Python, SQLite)의 로컬 보존이 필요한 상태이므로, 이에 대한 사용자 피드백을 수렴하여 삭제 범위를 결정할 예정입니다.
- 시스템 전체의 구조와 데이터 연동 플랜을 정리하는 `architecture.md` 문서를 한국어로 작성하기로 결정하였습니다.

### 데이터 수집 범위 확장 및 수집량 증폭 결정 (2026-06-06)
- 수집되는 경매 자산 데이터의 수가 너무 적은 원인을 분석한 결과, `court_scraper.py` 내부의 법원 수(10개), 기일 세션 수(15개), 검색 달(2개월)이 하드코딩으로 강하게 제한되어 있었음을 식별하였습니다.
- 이를 해결하기 위해 법원 전수 조사, 3달치 수집, 기일 세션 제한 전면 해제를 적용하여 데이터 수집량을 대폭 증대시키기로 결정하였습니다.
- 대법원 서버 차단 감지를 피하기 위해 요청 딜레이를 무작위로 유지하며 안정성을 확보하기로 결정하였습니다.

### 대시보드 탭 전환 시 화면 깨짐 및 페이지 누락 패치 결정 (2026-06-06)
- 탭 버튼 클릭 시 기존에 설정되어 있던 Tailwind 반응형 글자 크기 및 정렬 클래스가 `className` 재할당으로 인해 완전히 덮어써지면서 UI 찌그러짐 현상이 발생한 것을 식별하였습니다. 이를 보완하기 위해 `classList` API를 활용하여 활성/비활성 관련 테두리 및 텍스트 색상 클래스만 부분 변경하도록 결정하였습니다.
- 브라우저 해상도 변경 시 가동되는 `initResponsiveLayout` 함수가 메인 뷰들의 `className`을 재설정하면서, 탭 전환으로 부여된 `hidden` 클래스가 초기화되어 뷰들이 상호 충돌하여 화면에서 유실되거나 중첩 노출되는 문제를 확인하였습니다. 이를 해소하고자 전역 변수 `currentTabId`를 선언하고, 레이아웃 변경 직후 현재 탭 상태에 맞춰 각 뷰의 `hidden` 상태를 강제로 동기화하도록 구현하였습니다.

### 모바일 폰트 시인성 개선 및 Firestore Quota 초과 대처 (2026-06-06)
- 모바일 웹앱에서 필터 내부 글자 크기가 작다는 사용자 의견을 반영하여 `FeedScreen.tsx` 의 필터 패널 내 폰트 크기 설정을 2~3px 가량 전면 상향 조정하기로 하였습니다.
- 서버에 1,895건의 데이터가 있음에도 화면에 3개만 보인 원인을 추적한 결과, 무료 Firestore의 일일 읽기 한도(50,000 reads)가 소진되면서 Firestore SDK에서 `RESOURCE_EXHAUSTED` Quota 초과 에러가 발생한 것을 규명하였습니다.
- Firestore 로드 실패 시 예비용 하드코딩 데이터(`fallbackData` 4건 중 마감 필터로 3건 노출)만 반환되던 문제를 방어하기 위해, 데이터 로드 성공 시 브라우저 `localStorage`에 데이터를 캐싱해두고, 에러 시 이 캐시를 복구하여 1,895건의 실데이터를 무중단으로 볼 수 있도록 캐싱 전략을 결정하였습니다.

### 헤더 반응형 찌그러짐 및 계산기 키패드 시인성 개선 결정 (2026-06-08)
- 화면 폭이 좁은 모바일 뷰에서 헤더의 연동 센서 텍스트("실시간 데이터 연동 정상 (N건)")가 공간 부족으로 찌그러지고 위로 솟구치는 문제를 해결하기 위해, CSS 미디어 쿼리와 JS 갱신 부분에 반응형 클래스(`hidden sm:inline` 및 `inline sm:hidden`)를 도입하여 모바일에서는 "🟢 N건" 형태로 간소화해 노출하기로 하였습니다.
- 스마트 소요자금 계획서 가상 키패드의 전체 레이아웃 규격을 유지하면서 숫자 폰트 크기만 선택적으로 확대하기 위해, 숫자 버튼(클래스 `font-outfit`이 부여된 버튼들)에 한하여 폰트 크기를 데스크톱 기준 `16px`, 모바일 미디어 쿼리 기준 `18px`로 상향 적용하기로 결정하였습니다.

### 소셜 로그인 사전 구성(Authentication -> URL Configuration) 경고 팝업 추가 결정 (2026-06-08)

- 사용자가 이 사실을 쉽게 자각하고 자가 조치를 취할 수 있도록, 웹(index.html)과 모바일 앱(AuthScreen.tsx)의 소셜 로그인 버튼을 클릭했을 때 Supabase URL Configuration 설정 등록 여부를 묻는 사전 확인 대화상자(Confirm 및 Alert)를 강제 팝업하도록 조치하였습니다.
- 해당 팝업에서 사용자가 예를 클릭해야만 로그인이 속행되며, 등록되지 않았을 경우 사전에 차단되어 localhost:3000으로의 오작동 및 튕김 현상을 완벽하게 방지하였습니다.

### 대항력(대항권) 여부 정밀 진단 규칙 고도화 (2026-06-09)
- 사용자의 대항력(대항권) 추가 정밀 분석 요구를 반영하여, 기존 텍스트 기반 대항력 판단 로직을 세분화하여 4가지 분류로 고도화하였습니다.
- 첫째, 대항력 없음(안전)은 임차인이 없거나 대항력이 소멸되는 안전한 상태를 의미합니다.
- 둘째, 대항력 포기(안전)는 HUG 등 채권승계인의 대항력 포기 조건부 매각 건으로, 인수 금액이 발생하지 않는 안전한 상태를 뜻합니다.
- 셋째, 대항력 미상(확인 요망)은 임차인이나 전입세대는 존재하나 권리 정보가 불투명하여 현장 조사가 요망되는 상태를 뜻합니다.
- 넷째, 선순위 대항력 주의는 말소기준권리보다 빠른 선순위 임차인이 존재하여 보증금 인수 리스크가 확실한 위험 상태를 가리킵니다.
- 웹 대시보드(index.html)와 모바일 앱(DetailScreen.tsx)에 해당 규칙을 동시 적용하고, 대항력 미상 시 경고 뱃지 색상을 호박색(amber)으로 구분 노출되도록 뱃지 바인딩 스타일을 수정하여 프리미엄 권리진단 직관성을 향상시켰습니다.

### 자산 투자 성향(투자 가치형 / 실거주형) 구분 및 필터링 설계 (2026-06-09)
- 사용자의 경매 및 공매 매물 탐색을 고도화하기 위해 '투자형' 매물과 '실거주형' 매물을 직관적으로 분류하여 제공하기로 결정하였습니다.
- 분류 기준은 다음과 같습니다.
  - 상업용 부동산(상가/근린/상업/빌딩 등) 및 개발 투자성이 강한 토지/공장 등은 '투자형'으로 분류합니다.
  - 아파트/빌라/단독주택 등 주거용 부동산은 기본적으로 '실거주형'으로 분류합니다.
  - 단, 주거용 부동산 중 AI 권리진단 점수가 85점 이상인 A급 우량 매물은 '투자 & 실거주 겸용'으로 판별하여 양쪽 성향에 모두 매칭되며 뱃지도 동시에 노출되도록 처리합니다.
- 이에 따라 웹 대시보드와 모바일 앱에 투자 성향 필터(전체, 투자형, 실거주형)를 탑재하고 매물 카드 및 상세 뷰에 전용 뱃지(🏆 투자 추천, 🏠 실거주 추천)를 이식하기로 하였습니다.
